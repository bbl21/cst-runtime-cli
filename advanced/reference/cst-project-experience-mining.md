# CST 历史项目经验提炼设想

> 本文档用于记录一个后续方向：从历史 `CST` 项目中提炼可复用经验。  
> 它不是规则源，不改变当前正式生产链、P1 边界或现有 MCP/CLI 路线；规则仍以 `AGENTS.md` 为准，阶段计划仍以 `docs/project-goals-and-plan.md` 和 `advanced/planning/current-priority-checklist.md` 为准。

## 1. 设想定位

当前手头有不少历史 `CST` 项目，其中一些不是结构化优化任务记录，而是手工优化后保留下来的工程文件。

这批项目的潜在价值不是直接当成“训练数据”让大模型学会优化，而是把它们转成一个可检索、可比较、可追溯的经验库，用来辅助后续新任务判断。

更准确的定位是：

- 输入：历史 `CST` 工程文件，以及其中可能保留的 `1D Results`、`run_id`、`ModelHistory`、参数、材料、边界和监视器信息。
- 输出：设计经验、常见结构套路、可疑参数方向、常见权衡和失败线索。
- 用途：服务后续优化指导，不替代正式仿真与结果读取链路。

## 2. 当前已知前提

- 这些记录很多不是结构化 run 历史，而是保留在 `CST` 文件中的手工优化结果。
- 一部分项目里，某些 `1D` 结果可能保留在不同 `run_id` 中。
- 因此，可恢复的信息层级可能高于“只有最终模型”，但低于“完整参数优化日志”。

## 3. 这批项目可能提供什么

### 3.1 最终设计信息

可以尝试提取：

- 最终参数值
- 几何组成
- 材料选择
- 边界、端口、监视器
- 频段与结果树结构

### 3.2 部分历史结果信息

如果项目内还保留了不同 `run_id` 的 `1D Results`，可以尝试恢复：

- `run_id` 列表
- 每个 `run_id` 对应的 `S11`、`VSWR` 等 `1D` 曲线
- 某些结果序列上的改善/退化趋势

### 3.3 建模与调参痕迹

如果 `ModelHistory` 仍可读，可以提取：

- 建模构成顺序
- 常见布尔关系
- 参数命名习惯
- 哪些尺寸或结构明显是人工重点调过的

## 4. 明确限制

- 不能默认假设能恢复完整优化轨迹。
- 不能默认假设每个 `run_id` 都能绑定到完整参数快照。
- 不能把仅有的最终工程状态误写成“优化经验已验证因果关系”。
- 若只有 `Abs(E)` 或非真实增益口径，不能冒充 `gain/dBi` 经验。
- 这批项目更适合提炼“设计经验库”，不适合直接当成“优化策略训练集”。

## 5. 可行目标

更现实的目标是三层：

1. `项目摘要层`
   对单个历史 `CST` 项目提取统一摘要：参数、结构、材料、结果树、可见 `run_id`、可导出的 `1D` 结果。
2. `经验卡片层`
   让大模型基于多个项目摘要，总结短经验卡片，但每条卡片都必须能追溯到来源项目。
3. `检索辅助层`
   新任务来时，先检索相似历史项目，再输出可复用经验、已知风险和建议先试方向。

## 6. 第一阶段应先验证什么

在进入大规模整理前，先做小样本验证。建议选 `1` 到 `2` 个你确认保留了多个 `run_id` 的历史项目，只验证下面几件事：

- 能否稳定列出 `run_id`
- 能否稳定读出每个 `run_id` 的 `1D Results`
- 能否读到对应参数，或者至少读到最终参数集
- 能否区分“最终状态信息”和“历史结果信息”

如果第三项拿不到，就把这条路线定位成“历史结果经验库”；如果第三项能部分拿到，才值得进一步做“参数-结果经验库”。

## 7. 暂不并入当前主线

这个设想目前属于后续观察项，不应自动并入当前主线开发。它与当前生产链、Check Solid、runtime/CLI、正式 run 验证是不同层次的工作。

后续若真正启动，建议先补一份“单个历史 `CST` 项目摘要 schema”，明确哪些字段是可稳定抽取的，哪些只是可选证据。
