## 背景

经过真实仿真测试和架构对比，确认了 Skill vs prototype_optimizer 的功能重叠问题：

| 功能 | Skill (`scripts/`) | prototype_optimizer (`core/`) | 决策 |
|------|-------------------|------------------------------|------|
| 采样 | `sampler.py` | `core/sampler.py` | prototype_optimizer 改用 Skill |
| 评估 | `evaluator.py` | `core/evaluator.py` | prototype_optimizer 改用 Skill |
| 评分 | `scorer.py` | `core/scorer.py` | prototype_optimizer 改用 Skill |
| 仿真执行 | `optimizer_loop.py` | `CSTAdapter` | **废弃 CSTAdapter** |

---

## 架构决策

### 原则

- **LLM 适合**：参数空间搜索策略、步长选择、敏感度判断、异常处理决策
- **SKILL.md 适合**：描述流程、边界条件、注意事项（Human-readable）
- **scripts/ 适合**：提供可复用函数，LLM 按需调用（非强制循环）

### 划清界限

```
┌─────────────────────────────────────────────────────────────┐
│  Skill System (cst-antenna-optimizer)                       │
│  用途: LLM Agent 执行仿真优化                                │
├─────────────────────────────────────────────────────────────┤
│  SKILL.md          - 流程文档（直接读，不用执行脚本）        │
│  scripts/          - 可选工具函数（非必须）                  │
│  不做:              - 固定脚本循环                           │
└─────────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────────┐
│  prototype_optimizer                                        │
│  用途: 人类用户通过 UI 启动任务、查看历史、生成报告          │
├─────────────────────────────────────────────────────────────┤
│  app.py             - Streamlit UI                         │
│  storage/           - SQLite 历史管理                       │
│  core/              - 报告生成、RunWorkspace                │
│  不做:              - 仿真执行（委托给 Skill）             │
│  不做:              - evaluator/scorer/sampler（复用 Skill）│
└─────────────────────────────────────────────────────────────┘
```

### Task H: 原型优化器 UI 改造（进行中）

**目标**: 人类用户在 Streamlit UI 启动任务后，实际由 Skill 执行仿真

**当前架构问题**:
- `orchestrator.py` 原本依赖 `CSTAdapter`（封装双 COM session 流程）
- 删除 `CSTAdapter` 后，orchestrator 成为"过渡期遗留代码"

**SKILL.md 驱动的新架构**:

```
┌─────────────────┐      ┌─────────────────┐
│  Streamlit UI   │ IPC  │   LLM Agent     │
│  (prototype_    │ ───→ │  (按 SKILL.md   │
│   optimizer)    │      │   执行仿真)      │
└─────────────────┘      └─────────────────┘
        ↓                         ↓
   任务配置                  MCP Tools
   历史存储                  (modeler/
   报告生成                  results)
```

**IPC 设计选项**:

| 选项 | 说明 | 复杂度 |
|------|------|--------|
| A. LLM 作为独立 Agent | UI 提交任务 → LLM Agent 执行 SKILL.md → 结果回传 | 高 |
| B. 消息队列 | UI → Queue → LLM Worker → 结果 | 中 |
| C. Streamlit Callback | UI 触发后台线程执行 SKILL.md | 低 |

**推荐选项 A**：将 LLM Agent 定位为 "Skill 执行器"，UI 只负责：
1. 接收任务配置
2. 创建工作空间（RunWorkspace）
3. 提交给 LLM 执行
4. 存储历史和报告

**下一步实现**:
1. 在 `app.py` 中添加"LLM Agent 模式"按钮
2. LLM Agent 通过 MCP tools 直接执行 SKILL.md 流程
3. `orchestrator.py` 保留用于历史/报告，不参与仿真执行
