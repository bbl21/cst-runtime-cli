# 2026-04-23 Trae CLI 使用反馈分流

> 本文记录 Trae 完成 ref_0 CLI 低上下文验证后的使用反馈分流。  
> 当前主线仍以 `docs/current-priority-checklist.md` 为准；本文不改变 P0 门控顺序。

## 结论

Trae 已经证明“能跑通”，但反馈说明 `cst_runtime` 对低上下文 agent 还不够友好。当前主要矛盾已经从“能不能执行”转为“出错时能不能靠工具返回和文档自行定位、恢复并继续”。

本反馈全部接受，但不一次性展开实现。处理顺序如下：

1. `fresh-session` 真实 CST 远场导出/读取验证已完成，记录见 [`validations/2026-04-23-ref0-fresh-session-farfield-validation.md`](2026-04-23-ref0-fresh-session-farfield-validation.md)。
2. P0 已通过后，优先做 CLI 低上下文可用性加固，再进入 P1 优化指导原型。
3. 结果可视化、批量比较、性能和并行处理进入 P1/P2，不回填到 P0。

## 反馈分流

| 类别 | 反馈要点 | 分流 | 处理原则 |
| --- | --- | --- | --- |
| 参数格式和一致性 | `task_path` / `task_dir`、`output_files_json` / `output_files` 等命名不一致；参数错误提示不清晰 | P0 后加固 | 建立每个工具的 canonical 参数名；必要时提供兼容 alias；错误 JSON 必须返回未知字段、缺失字段、期望格式和最小示例 |
| 错误处理和用户反馈 | CST 连接、权限不足、文件锁定等错误缺少定位信息；长任务缺少进度反馈 | P0 门控相关 + P0 后加固 | 与 fresh-session 远场验证直接相关的连接、锁、权限问题先补；仿真进度反馈作为后续工具体验加固 |
| 流程集成和自动化 | 手动轮询仿真状态；参数修改、仿真、结果获取、状态更新不够顺滑 | P0 后加固 | 保留小粒度工具，同时增加可审计的阶段级编排命令；不能把流程封成不可诊断黑盒 |
| 权限和可靠性 | CST 进程清理经常遇到 `Access is denied`；项目锁定检测和容错不足 | P0 门控相关 | 继续使用强杀白名单；Access denied 时必须记录 PID、进程名、锁文件扫描结果和是否非阻塞；不得声称已杀掉 |
| 文档和使用指南 | 参数说明、端到端示例、最佳实践、故障排除不足 | P0 后加固 | 将 `describe-tool`、`args-template`、端到端示例和 troubleshooting 串成低上下文 agent 使用指南 |
| 结果管理和分析 | 文件命名元数据、可视化、批量比较能力不足 | P1/P2 | 不阻塞 P0；后续与优化指导、结果展示一起规划 |
| 性能优化 | CST 启动慢、内存占用、并行处理不足 | P2 | 只有在稳定性和可诊断性完成后再优化；避免为并行引入新的 session/锁风险 |

## 当前必须盯住的问题

### P0 门控相关

- `fresh-session` 真实 CST 远场导出/读取验证已完成；后续重点转为连接诊断、锁检测和权限残留的可读性加固。
- CST 连接失败时，错误必须能回答：连接哪个工程、工程是否已关闭、是否有 `.lok`、当前可见 CST 进程是什么、是否属于强杀白名单。
- 进程清理失败时，必须区分“未清理但不阻塞”和“仍有锁或占用导致阻塞”。
- 远场验证不得因为 CLI 体验优化而改写生产链路语义：真实 gain/dBi、导出路径、`close(save=False)` 仍是红线。

### P0 后加固

- 统一 CLI 参数命名表，写清 canonical name 和兼容 alias。
- 改进参数验证错误：返回结构化 JSON，不只返回 argparse 文本。
- 为低上下文 agent 增加端到端范例：准备 run、改参数、仿真、刷新 results、导出 S11、生成对比、清理 CST。
- 给常见失败增加 troubleshooting：CST 未启动、工程锁、权限拒绝、results 未刷新、JSON 格式错误。
- 考虑增加阶段级命令，但必须保留小粒度工具和完整 `flow_log`。

### P1/P2 观察项

- 结果文件命名加入参数值、run_id、时间戳等元数据。
- 增强 HTML 可视化和多组参数批量对比。
- 分析 CST 启动速度与重复连接开销。
- 评估可并行的只读分析，不并行写同一 CST 工程。

## 验收标准

后续处理这批反馈时，至少满足以下标准才算闭环：

- 任一工具参数写错时，返回 JSON 明确列出错误字段、期望字段和可直接修改的示例。
- `describe-tool` 与 `args-template` 对同一工具的参数名一致。
- Access denied 残留必须记录 PID、进程名、原因、锁文件扫描结果和阻塞判断。
- 低上下文 agent 可以只看 Skill、`usage-guide`、`list-pipelines`、`describe-pipeline`、`args-template` 和 troubleshooting 完成一次标准流程。
- 自动化程度提升不能牺牲可审计性；阶段级命令必须写入关键调用摘要、输入参数、输出路径、状态和错误。
