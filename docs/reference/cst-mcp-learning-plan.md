# CST_MCP 学习知识清单与阶段计划

> 本文档用于把个人学习计划绑定到 `CST_MCP` 的真实工程问题上。  
> 它不是规则源，不改变当前生产链、P0/P1 门控或工具路线；规则仍以 `AGENTS.md` 为准，阶段计划仍以 `docs/project-goals-and-plan.md` 和 `docs/current-priority-checklist.md` 为准。

## 1. 定位

当前最需要补的不是“完整计算机科班体系”，而是能直接提升项目把控能力的基础知识：

- 能看懂进程、文件锁、stdout/stderr、退出码、管道和缓存问题。
- 能把“脚本跑通”转化为“可复用、可验收、可追溯”的工程能力。
- 能设计和审查 `task/run/status/log/artifact` 的状态结构。
- 能判断工具接口是否适合低上下文 agent 调用。
- 能区分执行逻辑、决策逻辑、适配层和项目规则。

核心目标不是考试，而是让后续每次项目判断更稳：什么时候该 `blocked`，什么时候是 `needs_validation`，什么时候才允许 `validated`。

## 2. 知识清单

### 2.1 操作系统与命令行基础

必须掌握：

- 进程、线程、子进程、后台服务的区别。
- 文件锁、句柄、进程生命周期之间的关系。
- stdin、stdout、stderr、exit code 的职责。
- 管道如何传递数据，为什么 stdout 必须保持机器可读。
- Windows 下 `Access is denied`、残留进程、`.lok` 文件的基本含义。

对应项目问题：

- 为什么 CST 项目关闭后仍可能有残留进程。
- 为什么不能把 `Stop-Process` 失败写成“已清理”。
- 为什么 `cst_runtime` 的 stdout 要输出结构化 JSON。
- 为什么 PowerShell 内联 JSON 容易出错，`--args-file` 更稳。

验收标准：

- 能解释 `cleanup-cst-processes` 的结果为什么可能是“非阻塞残留”。
- 能说明 `stdout`、`stderr` 和日志文件各自该放什么。
- 能判断一个 CLI 工具是否会破坏管道调用。

### 2.2 软件测试与验收

必须掌握：

- happy path、error path、boundary case。
- 单元测试、集成测试、端到端验收的差异。
- 回归测试、错误分支、资源释放验证。
- 验收标准和测试证据的关系。
- demo、prototype、production-ready 的边界。

对应项目问题：

- 为什么“文件存在”不等于“结果正确”。
- 为什么工具返回 `success` 仍要检查输出文件、run_id、参数组合、单位和日志。
- 为什么 `needs_validation` 不能因为代码写完就改成 `validated`。

验收标准：

- 能为一个工具写出输入、输出、错误分支、资源释放、日志落盘的验收清单。
- 能审查 agent 的执行报告，指出缺少哪些证据。
- 能说明 workaround 什么时候可以临时接受，以及必须记录哪些移除条件。

### 2.3 数据建模与状态管理

必须掌握：

- entity、relationship、schema、invariant。
- 状态机与状态转换条件。
- append-only log、audit trail、artifact。
- 结构化状态文件与人工总结的区别。

对应项目问题：

- `task -> run -> stage -> artifact -> status` 的关系。
- `blocked`、`needs_validation`、`validated` 的转换条件。
- 为什么 `logs/tool_calls.jsonl`、`stages/*.json`、`status.json` 不能互相替代。
- 为什么 run 产物必须可追溯。

验收标准：

- 能画出当前 run 结构的数据关系。
- 能定义一个新阶段的状态字段和产物字段。
- 能发现状态文件、日志和实际输出之间的不一致。

### 2.4 CLI、协议与工具接口

必须掌握：

- CLI 参数、args file、JSON schema。
- 机器可读错误：`error_type`、`status`、`next_action`。
- client/server、adapter、protocol contract 的基本概念。
- 低上下文 agent 为什么需要自发现入口。

对应项目问题：

- 为什么 `doctor -> usage-guide -> list-tools -> describe-tool -> args-template` 是低上下文入口。
- 为什么 `invalid_json_args`、`cli_usage_error` 不能只是 argparse 文本。
- 为什么工具返回不能只有 `success`，还要包含输入、输出、文件、日志、错误分类。

验收标准：

- 能设计一个 agent 友好的工具返回结构。
- 能说明 `--args-file`、`--args-json`、`--args-stdin` 的适用边界。
- 能判断一个新 CLI 能力是否破坏现有契约。

### 2.5 数据结构与算法

必须掌握：

- list、dict、set、queue 的基本用途。
- tree、graph 的基本概念。
- 时间复杂度和组合爆炸。
- 穷举搜索、剪枝、缓存、粗扫与细扫。

对应项目问题：

- 多参数扫描的组合数量如何估算。
- 为什么黑盒 CST 仿真不能随便套“梯度下降”。
- 如何设计低成本参数探索：粗扫、局部细扫、回退、记录最优。
- 如何组织多个 run 的指标比较。

验收标准：

- 能估算一个参数扫描计划的运行成本。
- 能指出一个优化方案是否会组合爆炸。
- 能提出比全量穷举更节省的探索策略。

### 2.6 模块边界与架构

必须掌握：

- domain logic、adapter、orchestration、decision layer。
- 执行逻辑与决策逻辑分离。
- 稳定接口与临时脚本的差异。
- 规则、流程、背景说明、长期共识的文档分层。

对应项目问题：

- 什么能力该进入 `cst_runtime`。
- 什么判断应保留在 Skill。
- 什么经验应进入 docs。
- 什么红线才应进入 `AGENTS.md`。
- 为什么 CLI 是调用界面和 adapter，不应变成第二条生产链。

验收标准：

- 能判断一个新能力的落点：工具、runtime、Skill、docs、计划还是规则。
- 能解释选择理由，而不是只说“喜欢 CLI”或“不喜欢 MCP”。
- 能拒绝污染生产链的快速实现，并给出可验证替代路径。

## 3. 8 周学习计划

### 第 1-2 周：操作系统与命令行基础

学习目标：

- 弄清进程、文件锁、管道、stdout/stderr、exit code。
- 能解释 CST 进程残留、`.lok` 和 `Access is denied`。

每日任务：

- 每天学习一个概念，写 5-10 行解释。
- 找一个 `CST_MCP` 中的对应例子，说明这个概念如何影响项目判断。

项目练习：

- 写一页笔记：`为什么 CST 清理不能只说已结束`。
- 分析一次 `cleanup-cst-processes` 应该返回哪些字段。

阶段验收：

- 能把“进程未杀掉但无锁文件”准确归类为非阻塞残留或阻塞项。
- 能说明为什么 JSON stdout 被污染会影响低上下文 agent。

### 第 3 周：软件测试与验收

学习目标：

- 把“跑通”拆成可验收证据。
- 理解 happy path、错误分支、边界条件和回归测试。

每日任务：

- 每天选一个工具或流程，写出至少 3 个测试场景。
- 标出每个场景的预期输出、错误分类和停止条件。

项目练习：

- 为一个 `cst_runtime` 工具写验收清单。
- 把“文件存在”扩展成“路径、内容、run_id、参数、单位、日志均一致”。

阶段验收：

- 能审查 agent 报告，指出是否缺少错误分支、资源释放或产物核验。
- 能区分 demo、prototype、production-ready。

### 第 4 周：数据建模与状态管理

学习目标：

- 理解 `task/run/status/log/artifact` 是状态模型，不只是目录习惯。
- 能定义状态转换和审计记录。

每日任务：

- 画一个小模型：实体、字段、关系、状态。
- 说明哪些字段是事实，哪些字段是总结。

项目练习：

- 画出 `task -> run -> stage -> artifact -> status`。
- 定义 `blocked`、`needs_validation`、`validated` 的转换条件。

阶段验收：

- 能发现状态文件、日志、输出文件之间的不一致。
- 能说明为什么不能只靠人工总结判断任务完成。

### 第 5 周：CLI、协议与工具接口

学习目标：

- 理解 CLI 合同、JSON schema、错误协议和低上下文工具发现。
- 能设计 agent 可调用的工具接口。

每日任务：

- 每天读一个 CLI 返回，拆出 `status`、`error_type`、输入、输出、文件路径。
- 写出这个返回对下一步决策是否足够。

项目练习：

- 设计一个工具返回模板：`status/error_type/input/output/files/logs/next_action`。
- 解释 `doctor`、`usage-guide`、`describe-tool`、`args-template` 的职责。

阶段验收：

- 能判断一个 CLI 工具是否适合其他 agent 在低上下文下使用。
- 能说明什么时候使用 `--args-file`，什么时候才使用 `--args-stdin`。

### 第 6 周：数据结构与算法

学习目标：

- 能估算参数扫描成本。
- 能设计比全量穷举更节省的探索策略。

每日任务：

- 每天练一个复杂度估算。
- 把一个参数探索问题写成候选集合、评价函数和停止条件。

项目练习：

- 设计一个三参数扫描计划，说明为什么不全量穷举。
- 写出粗扫、局部细扫、回退、记录最优的流程。

阶段验收：

- 能看到组合爆炸风险。
- 能说明黑盒 CST 仿真为什么不能直接假设可用梯度下降。

### 第 7 周：模块边界与架构

学习目标：

- 理解执行层、决策层、适配层、规则层的分工。
- 能判断新能力应该落在哪里。

每日任务：

- 每天选一个项目能力，判断它属于执行、决策、状态、规则还是背景经验。
- 写出落点选择的理由。

项目练习：

- 对一个假设新能力做归类：工具、runtime、Skill、docs、计划、规则。
- 写出“为什么不直接写成临时脚本进生产链”。

阶段验收：

- 能用职责、验证成本、迁移边界判断架构，而不是用个人偏好判断。
- 能识别第二生产链风险。

### 第 8 周：综合演练

学习目标：

- 把前 7 周知识用于一次项目审查。

项目练习：

给一个假任务：新建 run、改参数、跑仿真、读 S11、生成对比、清理进程。输出：

- 执行计划。
- 验收清单。
- 错误分支。
- 状态转换。
- 产物列表。
- 是否允许 `validated` 的判断。

阶段验收：

- 能像负责人一样审查 agent 的执行报告。
- 能指出缺少的证据和下一步补验动作。

## 4. 每周节奏

建议每周只打一个主题，不同时补太多课程。

- 周一到周三：学习概念，每天 45-60 分钟。
- 周四：找 `CST_MCP` 中的对应例子。
- 周五：写一页项目化总结。
- 周末：做 30 分钟自测，回答“这个概念能解释项目里的哪个坑”。

每周总结模板：

```markdown
## 本周主题

## 学到的核心概念

## 对应的 CST_MCP 问题

## 我现在能做出的新判断

## 仍然不懂的问题

## 下周要验证的点
```

## 5. 阶段优先级

必须先补：

1. 操作系统基础。
2. 软件测试与验收。
3. 数据建模与状态管理。
4. CLI、协议与工具接口。

第二阶段再补：

1. 数据结构与算法。
2. 网络基础。
3. 架构设计。

暂缓：

- 编译原理。
- 分布式系统。
- 大规模数据库优化。
- 深度学习系统。
- 复杂优化算法。

这些后续有价值，但当前不能直接解决 `CST_MCP` 的把控问题。

## 6. 自测题

每两周回答一次，不查文档。

1. 一个 CST 进程无法强杀，但项目已关闭、无 `.lok`，应如何记录状态？
2. 一个 JSON 文件存在，为什么还不能证明结果正确？
3. 一个 CLI 工具把 warning 打到 stdout，会破坏什么？
4. 一个工具只返回 `success=true`，为什么不足以给 agent 决策？
5. 一个临时脚本跑通了，进入生产链前还缺什么？
6. 一个参数扫描有 5 个参数、每个 8 个候选，穷举成本是多少？如何降低？
7. 什么情况下任务只能标 `needs_validation`，不能标 `validated`？
8. 一个新经验应该进入 `AGENTS.md`、Skill、docs 还是 MEMORY，判断标准是什么？
9. 如何区分 demo、prototype、production-ready？
10. 如何判断一个新入口是合理 adapter，还是第二条生产链风险？

## 7. 完成标准

8 周后应达到以下状态：

- 能解释一个失败属于进程、锁、缓存、接口、数据结构还是业务规则问题。
- 能把“跑通”拆成可验收证据。
- 能设计低上下文 agent 可用的工具接口。
- 能判断一个能力应该沉淀到工具、Skill、docs 还是计划。
- 能拒绝看似快速但会污染生产链的做法，并给出可执行替代方案。

若仍做不到以上五点，应继续重复第 1-5 周，不急于进入复杂算法和系统架构学习。
