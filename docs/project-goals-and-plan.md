# 项目目标与计划

> 本文档用于定义项目定位、阶段目标、执行计划，以及目标与计划的维护规则。  
> 它不是规则源；全局规则与红线以 `AGENTS.md` 为准，执行流程以 `skills/cst-simulation-optimization/SKILL.md` 为准。

## 1. 项目定位

本项目当前定位为：

**一个面向 LLM 主导、人工监督的 CST 参数化优化执行系统。**

它当前重点解决的是：

- 基于既有参数化 CST 模型开展优化任务
- 通过统一执行链完成仿真、结果读取、导出、对比、记录与知识回收
- 将稳定重复劳动沉淀为程序模块与 MCP 工具
- 将灵活决策与经验性判断交给 LLM + Skill

本项目长期愿景是：

**逐步演进为“自然语言需求 -> 原型生成 -> 仿真优化 -> 结果展示”的闭环系统。**

但在当前阶段，项目不以“全自动 3D 建模闭环”作为近期验收目标。

## 2. 核心分工原则

### 2.1 LLM 负责灵活部分

以下内容优先交给 LLM + Skill：

- 优化目标理解与拆解
- 参数探索方向选择
- 实验结果解释
- 异常现象分析与下一步判断
- 经验总结与知识分流

### 2.2 程序与 MCP 负责重复部分

以下内容优先沉淀为程序模块或 MCP 工具：

- 结果读取与格式转换
- 指标计算
- 导出能力
- 对比与展示
- 任务状态落盘
- run 目录管理
- 稳定、可复用的执行接口

### 2.3 沉淀原则

- 已经稳定、重复出现、适合复用的能力，应优先沉淀到程序模块或 `mcp/*.py`
- `SKILL.md` 负责流程、策略、经验和失败恢复，不承担 Python 代码复用层职责
- 一次性探针、验证脚本可暂存于 `tmp/` 或 `tools/`，但不应长期充当正式链路
- 工具粒度按判断需求划分：固定、长流程、机械重复的步骤可以封装为阶段级工具或编排命令；需要 agent 依据返回信息判断下一步的步骤，应保留小粒度工具，并返回结构化指标、状态、错误和建议动作
- 生产自动化不是把整条优化链封成黑盒，而是让 agent 在 Skill 约束下调用工具、读取返回、判断是否继续、重试、换策略或询问用户

## 3. 阶段目标

## 3.1 近期目标

- 状态：`active`
- 时间范围：2026-04-17 起未来一周
- 目标：
  - 把当前生产链收口为一个可迁移、可复用、不依赖上下文和特定 coding agent 的正式执行入口
  - 去除对临时脚本、旁路流程和隐式上下文的生产依赖
  - 将 MCP、Skill、知识系统、计划系统整合到同一主链路中
  - 在不破坏主线的前提下，为“优化指导功能”预留可接入边界

### 当前状态说明

- 上一轮“真实优化任务 + 结果展示 + 可对外讲述”的近期计划已完成，并作为已验证阶段成果保留
- 当前新一轮近期开工项已重置，主攻方向从“做出一个可信演示案例”切换为“收口正式生产链”
- 本轮窗口只有一周，基础目标必须优先于进阶目标；进阶目标只能在基础目标不失控的前提下进入

### 近期验收标准

- 存在且只存在一个正式交互入口，能驱动当前主生产流程
- 从最小上下文出发，不依赖特定 coding agent 记忆或临时手工补洞，仍能完成完整生产流程
- 当前生产流程不再依赖 `tmp/`、一次性脚本或旁路流程作为默认入口
- MCP、Skill、知识系统、计划系统在主链路中职责清晰，且能通过正式入口被调用或约束
- 至少完成一次端到端验证，产物仍按标准 `tasks/task_xxx/runs/run_xxx/` 结构落盘
- 若进入进阶目标，必须产出一个可演示的优化指导原型：能够基于建模代码和用户补充信息形成面向天线任务的优化建议

### 近期主要风险

- 为了赶一周窗口，容易把“统一入口”做成新的包装层，而底层仍依赖旧旁路链
- 进阶目标若过早并入主线，会挤压基础目标时间，导致两边都不稳定
- MCP、Skill、知识系统、计划系统若只是表面串联，而不是职责收口，会形成新的耦合和上下文依赖
- 若验证只在当前熟悉上下文中成立，仍无法证明其可迁移性

### 近期明确不做

- 不把“从自然语言直接生成 3D 模型”作为本轮目标
- 不扩展到多种器件类型的全覆盖
- 不为追求好看而重做大规模 UI
- 不并行推进与当前生产链收口无关的高不确定研究方向
- 不在基础目标未完成前，把论文研读和优化指导能力变成新的并行主线

## 3.2 中期目标

- 状态：`planned`
- 时间范围：未来数周到数月
- 目标：
  - 完成这套系统的产品级工程实现
  - 让系统可投入真实生产任务
  - 在工程体系或方法论上形成一定的学术创新
  - 为学术会议投稿准备可复现内容

### 中期验收标准

- 形成稳定统一的正式执行链，避免多条并行旧链路
- 关键能力具备明确输入输出、错误处理和状态管理
- 可连续完成多次真实优化任务，而不是一次性演示
- 形成至少一个可讲清楚、可实验验证的创新点

### 中期主要风险

- “产品化”和“论文化”节奏混杂，互相干扰
- 系统工程部分不够稳，导致无法承载真实生产使用
- 创新点表述空泛，停留在愿景层而非可验证层

### 中期明确不做

- 不做通用 CST GUI 替代品
- 不做覆盖所有仿真对象和全部工作流的大平台
- 不为了论文包装而牺牲系统可维护性

## 3.3 长期目标

- 状态：`planned`
- 时间范围：长期演进方向
- 目标：
  - 实现自然语言需求到原型生成、仿真优化、结果展示的闭环系统

### 长期成功图景

- 用户输入需求，例如某类天线、滤波器或其他电磁对象及其性能指标
- 系统能完成资料检索、原型选择或生成、建模、仿真优化和结果展示
- 人类更多负责监督、校验与决策确认，而不是手工推动每个执行环节

### 长期主要约束

- LLM 当前在复杂 3D 空间建模上的稳定性仍不足
- 建模示教成本高，且鲁棒性不够
- 该阶段能力依赖未来更强模型或外部能力接入，例如几何/建模辅助系统

### 长期明确不做

- 不承诺在当前阶段直接实现全自动建模闭环
- 不为了追赶长期愿景而破坏当前优化底座建设

## 4. 当前阶段主攻方向

当前阶段只集中解决一件事：

**把现有生产链做成可迁移、单入口、可复用的正式执行底座，并在其上预留优化指导能力的接入口。**

当前主攻方向包括：

- 收口唯一正式交互入口
- 清理临时脚本和旁路流程的生产依赖
- 让 MCP、Skill、知识系统、计划系统在同一链路内协同工作
- 验证在低上下文条件下仍可稳定执行
- 只在底座稳定后，接入优化指导原型

当前暂缓方向包括：

- 全自动 3D 建模
- 通用化器件全覆盖
- 与本轮正式入口无关的大规模产品界面重构
- 未经门控就并入主线的论文研读与优化指导扩展
- 高不确定、难以在一周窗口内验收的扩展能力

## 5.1 本周基础目标

本周基础目标只按一个顺序推进：

**先定义并冻结正式入口，再收口主链，再补齐四类系统协同，最后做低上下文验证与收尾。**

### 5.1.1 阶段 A：范围冻结与正式入口定义

- 时间建议：第 1 天
- 目标：
  - 盘点当前所有会实际触发生产流程的入口、脚本、文档约束和人工步骤
  - 明确“唯一正式交互入口”是什么，以及它在本项目中的唯一职责边界
  - 冻结本周范围，明确哪些旁路链和临时脚本本周必须退出生产职责
- 交付物：
  - 一份当前生产链入口清单
  - 一份正式入口的最小输入/输出/阶段职责定义
  - 一份本周明确停用、迁移或保留观察的旁路清单
- 完成信号：
  - 团队可以用一句话说清当前唯一正式入口是什么
  - run 创建、执行、导出、对比、状态落盘分别归哪个正式环节负责已经明确
  - 本周不再新增第二条“临时先跑起来”的并行主链

### 5.1.2 阶段 B：主生产链收口与旁路清理

- 时间建议：第 2 至第 3 天
- 目标：
  - 把 run 创建、仿真执行、results 刷新、结果导出、对比展示、状态落盘统一到正式主链
  - 将当前仍承担生产职责的 `tmp/`、一次性脚本、旧入口、手工补洞步骤迁回主链或降级为非生产路径
  - 明确哪些地方仍存在历史兼容入口，但禁止继续作为默认路径
- 交付物：
  - 一条可以从正式入口走到底的主生产链
  - 一份旁路清理结果：已迁移、已禁用、暂保留观察三类
  - 一份主链关键节点列表：每一步输入什么、输出什么、失败时去哪看
- 完成信号：
  - 不借助临时脚本也能完成主流程
  - 不依赖当前会话记忆也能知道下一步该调用哪个正式能力
  - 用户可见的生产路径只剩一条默认入口

### 5.1.3 阶段 C：四类系统协同补齐

- 时间建议：第 4 至第 5 天
- 目标：
  - 把 MCP、Skill、知识系统、计划系统在主链中的职责边界写清并落到实际流程
  - 避免“代码在这边，规则在那边，计划在第三处，但没有任何地方真正引用”的松散耦合
  - 把当前主线真正需要的知识回收点和计划维护点插回正式流程
- 交付物：
  - MCP 职责说明：哪些能力必须沉淀为工具
  - Skill 职责说明：哪些判断和流程约束保留在 Skill
  - 知识分流清单：规则、经验、长期共识分别落哪
  - 计划维护点：什么情况下更新 `project-goals-and-plan.md` 与 `current-priority-checklist.md`
- 完成信号：
  - 四类系统都能回答“为什么我在这里，而不是在别处”
  - 当前主链每个关键阶段都能找到对应的规则、计划或工具归属
  - 计划系统不再只是说明书，而能直接约束近期执行顺序

### 5.1.4 阶段 D：低上下文验证与收尾固化

- 时间建议：第 6 至第 7 天
- 前置条件：
  - CLI-first 架构门控已完成，且明确 CLI 不作为第二条生产链
  - 已确认正式入口仍是 `tasks/task_xxx_slug/`，核心能力位于 `cst_runtime/`
  - 已确认 MCP 只是兼容 adapter，不再作为新能力首要开发目标
  - 已确认 `prototype_optimizer` 不进入轻量主线和默认迁移包
- 目标：
  - 在最小必要上下文下做一次端到端验证，检查正式入口是否真可迁移
  - 验证标准 run 产物、状态文件、错误分支、知识回收链是否都能正常工作
  - 给出是否允许进入进阶目标的门控结论
- 交付物：
  - 一次低上下文端到端验证记录
  - 一份残留问题清单：阻塞项、非阻塞项、观察项
  - 一份门控结论：能否进入优化指导功能原型
- 完成信号：
  - 非当前对话发起者也能根据正式入口和文档理解流程并执行
  - `run_xxx/{projects,exports,logs,stages,analysis}` 产物完整可追溯
  - 能明确说出本周已解决了哪些结构性问题、还剩哪些不能带入进阶目标的风险

## 7.3 闭环系统成型

- 将需求理解、原型生成、仿真优化和结果展示逐步串联为完整闭环

## 8. 目标与计划维护规则

## 8.1 分层维护规则

- 近期目标驱动当下开发与任务选择
- 中期目标约束架构方向与工程沉淀方式
- 长期目标提供方向，不直接支配日常实现优先级

## 8.2 状态维护规则

每个目标至少维护以下字段：

- 状态
- 目标描述
- 验收标准
- 主要风险
- 明确不做内容

允许使用的状态：

- `active`
- `planned`
- `blocked`
- `validated`
- `archived`

## 8.3 复审节奏

- 近期目标：按天或按关键任务完成后复审
- 中期目标：按周复审
- 长期目标：按月或按关键技术条件变化复审

## 8.4 变更规则

- 长期愿景轻易不改，只在方向认知发生明显变化时调整
- 中期目标允许调整路径，但不轻易更换主题
- 近期计划可以按现实情况调整，但不能偏离中期主轴
- 新想法若不能直接支撑近期目标或中期主轴，不进入当前主线

## 8.5 插队控制规则

以下事项默认不得插队进入当前主线：

- 与近期展示无关的高不确定功能
- 只服务长期愿景、短期无法验收的探索
- 会引入旁路生产链路的临时实现
- 只为“看起来先进”而加入的新组件

## 8.6 知识回收规则

每次阶段任务结束后，应判断是否形成以下内容：

- 新的长期红线
- 新的稳定执行流程
- 新的建模/调试经验
- 新的跨任务共识

若答案为“是”，应按分层原则分流到：

- `AGENTS.md`：规则、红线、禁令
- `skills/.../SKILL.md`：执行流程、策略、经验与恢复方法
- `docs/`：背景、计划、说明、阶段总结
- `MEMORY.md`：稳定事实、长期共识、确认决策

## 8.7 当前最重要的维护纪律

**长期愿景不能直接支配近期开发；近期开发必须服务于中期平台化方向，而不是为了一次性演示堆积临时能力。**
