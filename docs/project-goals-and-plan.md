# 项目目标与计划

> 本文档用于定义项目定位、阶段目标、执行计划，以及目标与计划的维护规则。  
> 它不是规则源；全局规则与红线以 `AGENTS.md` 为准，执行流程以 `skills/cst-simulation-optimization/SKILL.md` 为准。

## 1. 项目定位

本项目当前定位为：

**一个面向 LLM 主导、人工监督的 CST 参数化优化执行系统。**

它当前重点解决的是：

- 基于既有参数化 CST 模型开展优化任务
- 通过统一执行链完成仿真、结果读取、导出、对比、记录与知识回收
- 将稳定重复劳动沉淀为程序模块与 MCP 工具
- 将灵活决策与经验性判断交给 LLM + Skill

本项目长期愿景是：

**逐步演进为“自然语言需求 -> 原型生成 -> 仿真优化 -> 结果展示”的闭环系统。**

但在当前阶段，项目不以“全自动 3D 建模闭环”作为近期验收目标。

## 2. 核心分工原则

### 2.1 LLM 负责灵活部分

以下内容优先交给 LLM + Skill：

- 优化目标理解与拆解
- 参数探索方向选择
- 实验结果解释
- 异常现象分析与下一步判断
- 经验总结与知识分流

### 2.2 程序与 MCP 负责重复部分

以下内容优先沉淀为程序模块或 MCP 工具：

- 结果读取与格式转换
- 指标计算
- 导出能力
- 对比与展示
- 任务状态落盘
- run 目录管理
- 稳定、可复用的执行接口

### 2.3 沉淀原则

- 已经稳定、重复出现、适合复用的能力，应优先沉淀到程序模块或 `mcp/*.py`
- `SKILL.md` 负责流程、策略、经验和失败恢复，不承担 Python 代码复用层职责
- 一次性探针、验证脚本可暂存于 `tmp/` 或 `tools/`，但不应长期充当正式链路

## 3. 阶段目标

## 3.1 近期目标

- 状态：`active`
- 时间范围：2026-04-17 起未来一周
- 目标：
  - 把当前生产链收口为一个可迁移、可复用、不依赖上下文和特定 coding agent 的正式执行入口
  - 去除对临时脚本、旁路流程和隐式上下文的生产依赖
  - 将 MCP、Skill、知识系统、计划系统整合到同一主链路中
  - 在不破坏主线的前提下，为“优化指导功能”预留可接入边界

### 当前状态说明

- 上一轮“真实优化任务 + 结果展示 + 可对外讲述”的近期计划已完成，并作为已验证阶段成果保留
- 当前新一轮近期开工项已重置，主攻方向从“做出一个可信演示案例”切换为“收口正式生产链”
- 本轮窗口只有一周，基础目标必须优先于进阶目标；进阶目标只能在基础目标不失控的前提下进入

### 近期验收标准

- 存在且只存在一个正式交互入口，能驱动当前主生产流程
- 从最小上下文出发，不依赖特定 coding agent 记忆或临时手工补洞，仍能完成完整生产流程
- 当前生产流程不再依赖 `tmp/`、一次性脚本或旁路流程作为默认入口
- MCP、Skill、知识系统、计划系统在主链路中职责清晰，且能通过正式入口被调用或约束
- 至少完成一次端到端验证，产物仍按标准 `tasks/task_xxx/runs/run_xxx/` 结构落盘
- 若进入进阶目标，必须产出一个可演示的优化指导原型：能够基于建模代码和用户补充信息形成面向天线任务的优化建议

### 近期主要风险

- 为了赶一周窗口，容易把“统一入口”做成新的包装层，而底层仍依赖旧旁路链
- 进阶目标若过早并入主线，会挤压基础目标时间，导致两边都不稳定
- MCP、Skill、知识系统、计划系统若只是表面串联，而不是职责收口，会形成新的耦合和上下文依赖
- 若验证只在当前熟悉上下文中成立，仍无法证明其可迁移性

### 近期明确不做

- 不把“从自然语言直接生成 3D 模型”作为本轮目标
- 不扩展到多种器件类型的全覆盖
- 不为追求好看而重做大规模 UI
- 不并行推进与当前生产链收口无关的高不确定研究方向
- 不在基础目标未完成前，把论文研读和优化指导能力变成新的并行主线

## 3.2 中期目标

- 状态：`planned`
- 时间范围：未来数周到数月
- 目标：
  - 完成这套系统的产品级工程实现
  - 让系统可投入真实生产任务
  - 在工程体系或方法论上形成一定的学术创新
  - 为学术会议投稿准备可复现内容

### 中期验收标准

- 形成稳定统一的正式执行链，避免多条并行旧链路
- 关键能力具备明确输入输出、错误处理和状态管理
- 可连续完成多次真实优化任务，而不是一次性演示
- 形成至少一个可讲清楚、可实验验证的创新点

### 中期主要风险

- “产品化”和“论文化”节奏混杂，互相干扰
- 系统工程部分不够稳，导致无法承载真实生产使用
- 创新点表述空泛，停留在愿景层而非可验证层

### 中期明确不做

- 不做通用 CST GUI 替代品
- 不做覆盖所有仿真对象和全部工作流的大平台
- 不为了论文包装而牺牲系统可维护性

## 3.3 长期目标

- 状态：`planned`
- 时间范围：长期演进方向
- 目标：
  - 实现自然语言需求到原型生成、仿真优化、结果展示的闭环系统

### 长期成功图景

- 用户输入需求，例如某类天线、滤波器或其他电磁对象及其性能指标
- 系统能完成资料检索、原型选择或生成、建模、仿真优化和结果展示
- 人类更多负责监督、校验与决策确认，而不是手工推动每个执行环节

### 长期主要约束

- LLM 当前在复杂 3D 空间建模上的稳定性仍不足
- 建模示教成本高，且鲁棒性不够
- 该阶段能力依赖未来更强模型或外部能力接入，例如几何/建模辅助系统

### 长期明确不做

- 不承诺在当前阶段直接实现全自动建模闭环
- 不为了追赶长期愿景而破坏当前优化底座建设

## 4. 当前阶段主攻方向

当前阶段只集中解决一件事：

**把现有生产链做成可迁移、单入口、可复用的正式执行底座，并在其上预留优化指导能力的接入口。**

当前主攻方向包括：

- 收口唯一正式交互入口
- 清理临时脚本和旁路流程的生产依赖
- 让 MCP、Skill、知识系统、计划系统在同一链路内协同工作
- 验证在低上下文条件下仍可稳定执行
- 只在底座稳定后，接入优化指导原型

当前暂缓方向包括：

- 全自动 3D 建模
- 通用化器件全覆盖
- 与本轮正式入口无关的大规模产品界面重构
- 未经门控就并入主线的论文研读与优化指导扩展
- 高不确定、难以在一周窗口内验收的扩展能力

## 5. 近期执行计划

当前执行清单见：`docs/current-priority-checklist.md`

当前状态：

- 上一轮近期执行计划已完成并验证
- 本轮进入新的“一周收口计划”
- 当前默认策略是：基础目标先打通，再决定是否进入进阶目标

## 5.1 本周基础目标

### P0：收口唯一正式交互入口

- 明确当前唯一正式入口的职责边界
- 保证 run 创建、执行、导出、对比、状态落盘从该入口进入
- 禁止把临时脚本、手工步骤或特定 agent 记忆当作隐含前置条件

### P0：统一生产链并清理旁路依赖

- 盘点现有 MCP、Skill、知识系统、计划系统之间的调用关系
- 将当前默认生产流程统一到同一正式链路
- 标记并移除不应继续承担生产职责的临时脚本、旧入口或旁路流程

### P0：补齐可迁移性约束

- 明确正式入口所需的最小输入、输出和上下文
- 让关键流程不依赖特定 coding agent 的会话记忆
- 明确错误分支、恢复方式和状态落盘规则

### P0：完成整体验证

- 从低上下文条件启动一次完整流程验证
- 确认标准 run 目录、导出结果、状态记录和知识回收链路都可正常工作
- 将本轮约束、边界和剩余风险同步回计划与相关知识文档

## 5.2 本周进阶目标（门控进入）

进入条件：

- 基础目标已完成，且正式生产链不再依赖临时脚本、旁路流程、特定 agent 或隐式上下文
- 当前正式入口、状态管理和知识约束已稳定可复用

进阶目标：

- 接入一个“优化指导功能”原型
- 能基于建模代码识别当前天线问题和已有实现信息
- 能主动询问用户补齐缺失的天线需求与约束
- 能围绕具体需求寻找并研读相关论文，形成结构化优化指导
- 输出的指导内容应服务于当前优化任务，而不是变成独立研究玩具

## 5.3 上一轮近期计划归档（已完成）

- 真实优化案例交付：已完成
- 基础结果展示：已完成
- 导师沟通材料整理：已完成
- 必要的不稳定点补强：已完成

## 5.4 近期交付物

- 一个唯一正式交互入口及其对应的正式生产链
- 一套不依赖临时脚本、旁路流程和特定 agent 的可迁移执行方式
- 一次完整的标准 run 验证产物
- 一份说明 MCP、Skill、知识系统、计划系统如何协同的结构化说明
- 若进入进阶目标，一份基于建模代码、用户信息和论文研读的优化指导原型输出

## 6. 中期建设计划

## 6.1 执行链统一

- 明确唯一正式生产链路
- 清理旁路执行方式和旧原型残留
- 保证不同入口最终汇合到同一执行链

## 6.2 工程化能力补全

- 完善任务状态管理
- 完善错误分支和恢复策略
- 完善结果导出与展示接口
- 完善可复用模块边界

## 6.3 生产化与复用

- 将稳定重复能力直接沉淀为 MCP 或程序模块
- 降低对一次性手工干预的依赖
- 提高多任务复用能力与结果可追溯性

## 6.4 学术创新提炼

- 从真实任务中提炼系统方法或流程创新点
- 形成可实验验证、可比较的创新表述
- 为会议投稿准备实验材料和系统描述

## 7. 长期演进路线

## 7.1 优化底座继续增强

- 继续增强参数化优化执行能力
- 扩展更多目标类型与更复杂的任务闭环

## 7.2 建模能力接入

- 关注更适合几何/空间建模的外部能力
- 在能力成熟时，以模块化方式接入原型生成或建模辅助系统
- 优先采用“辅助建模”而非一开始就追求完全自治建模

## 7.3 闭环系统成型

- 将需求理解、原型生成、仿真优化和结果展示逐步串联为完整闭环

## 8. 目标与计划维护规则

## 8.1 分层维护规则

- 近期目标驱动当下开发与任务选择
- 中期目标约束架构方向与工程沉淀方式
- 长期目标提供方向，不直接支配日常实现优先级

## 8.2 状态维护规则

每个目标至少维护以下字段：

- 状态
- 目标描述
- 验收标准
- 主要风险
- 明确不做内容

允许使用的状态：

- `active`
- `planned`
- `blocked`
- `validated`
- `archived`

## 8.3 复审节奏

- 近期目标：按天或按关键任务完成后复审
- 中期目标：按周复审
- 长期目标：按月或按关键技术条件变化复审

## 8.4 变更规则

- 长期愿景轻易不改，只在方向认知发生明显变化时调整
- 中期目标允许调整路径，但不轻易更换主题
- 近期计划可以按现实情况调整，但不能偏离中期主轴
- 新想法若不能直接支撑近期目标或中期主轴，不进入当前主线

## 8.5 插队控制规则

以下事项默认不得插队进入当前主线：

- 与近期展示无关的高不确定功能
- 只服务长期愿景、短期无法验收的探索
- 会引入旁路生产链路的临时实现
- 只为“看起来先进”而加入的新组件

## 8.6 知识回收规则

每次阶段任务结束后，应判断是否形成以下内容：

- 新的长期红线
- 新的稳定执行流程
- 新的建模/调试经验
- 新的跨任务共识

若答案为“是”，应按分层原则分流到：

- `AGENTS.md`：规则、红线、禁令
- `skills/.../SKILL.md`：执行流程、策略、经验与恢复方法
- `docs/`：背景、计划、说明、阶段总结
- `MEMORY.md`：稳定事实、长期共识、确认决策

## 8.7 当前最重要的维护纪律

**长期愿景不能直接支配近期开发；近期开发必须服务于中期平台化方向，而不是为了一次性演示堆积临时能力。**
